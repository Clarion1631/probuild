## Critical Bugs

### BUG-1: `/leads` page crashes with "Something went wrong" — Server Component error
**Severity:** CRITICAL
**Workflow:** W1 (Lead Creation)
**Tests failed:** W1.2 (directly), W1.3, W1.4, W1.5, W2.1, W2.2, W2.3, W3.1+ (10 tests blocked)

**What happened:**
Navigating to `/leads` renders the sidebar and layout correctly but the main content area displays:
> "Something went wrong — Could not load leads. Please try again."

Two Server Component render errors appear in the browser console:
```
Error: An error occurred in the Server Components render. The specific message is omitted
in production builds to avoid leaking sensitive details.
```

**Impact:**
- Lead creation is completely blocked
- Estimate and Invoice workflows (W2, W3) cannot run — they depend on a lead being created
- 10 downstream tests marked "did not run"

**Likely cause:**
A Server Component in `/leads` is throwing an unhandled exception. In production builds the error digest is suppressed. Check Sentry for the full stack trace around the time of this run:
```
sentry-cli issues list --org golden-touch-remodeling --project probuild --query "leads"
```

**Page still works:** `/leads` navigation loads (W1.1 passes), sidebar renders, no white-screen crash — only the leads list content throws. Likely a `db.lead.findMany()` or similar Prisma query failing.

### BUG-2: Clock-out on `/time-clock` triggers error toast
**Severity:** HIGH
**Workflow:** W4 (Time Clock & Expenses)
**Test failed:** W4.2

**What happened:**
1. Navigate to `/time-clock` — page loads cleanly (W4.1 passes)
2. Select project from dropdown, click "Clock In" — appears to succeed (status shows "Clocked In")
3. Wait 3 seconds
4. Click "Clock Out" — UI updates to "Clocked Out" but an error toast (`[role="alert"]`) fires immediately after

**Impact:**
The clock-out action partially fails. The UI state changes to "Clocked Out" but the server returned an error. Time entry may not be persisted, or it persisted but the response handler failed.

**Screenshots:** `w4_s2_clocked-out.png`, `w4_s2_before-clock-in.png`, `w4_s2_clocked-in.png` (in `qa-screenshots/`)

**Recommended fix:**
Check the `/api/time-clock` (or equivalent) endpoint for the clock-out POST. Look for missing required fields, Prisma constraint violations, or async timing issues (the entry may not exist server-side yet when clock-out fires if clock-in is async).

## Test Infrastructure Issue (Not an App Bug)

### W9.7: 25 project sub-pages — test timeout at 90s
**Severity:** Low (infrastructure)
**Impact:** Test result inconclusive — app not confirmed broken

The navigation audit for all 25 project sub-pages (overview, estimates, invoices, contracts, change-orders, purchase-orders, schedule, tasks, takeoffs, floor-plans, dailylogs, budget, costing, financial-overview, selections, mood-boards, messages, timeclock, time-expenses, files, bid-packages, retainers, settings, client-portal) exceeded the 90-second test timeout.

Each page requires `waitUntil: "networkidle"` (2-4s per page) × 25 pages = 50-100s. The browser context was closed by Playwright after timeout, causing the final screenshot to fail with `Target page, context or browser has been closed`.

**Fix:** Increase per-test timeout for W9.7 to 180s, or split into two tests of 12-13 pages each.
